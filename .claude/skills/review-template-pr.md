# Skill: /review-template-pr

MRM entry point for processing an incoming candidate template PR. Reads the PR via the GitHub MCP, posts the MRM Review Summary comment, and walks through each section requiring MRM attention.

**Audience:** MRM reviewers working in this repo. Not for data scientists.

---

## Step 1: Identify the PR

If the user invoked `/review-template-pr` without a PR number, ask:
> "Which PR should I load? (Provide the PR number, e.g., `42`)"

If a PR number was provided, proceed.

---

## Step 2: Read the PR via GitHub MCP

Use the GitHub MCP to fetch:
- PR title, body, author, created date
- Files changed (the candidate template)
- Existing comments

Parse the structured payload from the PR body. Look for these fields:
- Pattern
- Algorithm
- Requesting project / developer
- Tollgate date
- Section classification table (INHERITED / AMENDED / NET NEW counts and lists)

If the payload is missing or malformed, note this in the review summary and ask the user if they want to proceed with manual section identification.

---

## Step 3: Post the MRM Review Summary comment

Post a comment on the PR with this exact structure:

```
## MRM Review Summary

**Pattern:** [PATTERN]
**Algorithm:** [ALGORITHM]
**Requested by:** [DEVELOPER / PROJECT]
**Tollgate:** [DATE or "Not specified"]
**Received:** [PR created date]

### Section Breakdown

| Classification | Count | Sections |
|---|---|---|
| Inherited (no review needed) | [N] | [list or "—"] |
| Amended (review required) | [N] | [list] |
| Net New (review required) | [N] | [list] |

**Total MRM review scope:** [N] sections
**Estimated review effort:** [LOW (1–3 sections) / MEDIUM (4–7) / HIGH (8+)]

---
*Generated by Claude Code on [date]. MRM reviewer: self-assign this PR to claim it.*
```

After posting, confirm to the MRM reviewer: "Review summary posted. The PR is now ready for MRM assignment."

---

## Step 4: Load the candidate template

Read the candidate template file from the PR. Identify and extract:
- All AMENDED sections (marked `⚠️ AMENDED — MRM REVIEW REQUIRED`)
- All NET NEW sections (marked `⚠️ NET NEW — MRM REVIEW REQUIRED`)
- The inline change annotations (`<!-- CHANGED: ... -->`)

---

## Step 5: Walk through each section requiring review

For each AMENDED or NET NEW section, in order:

1. Present the section content to the MRM reviewer
2. For AMENDED sections: highlight the specific changes using the inline `<!-- CHANGED -->` annotations
3. For NET NEW sections: summarize what the section is trying to document and why it was added
4. Ask: *"Does this section accurately reflect the documentation requirements for this pattern/algorithm? Any edits needed, or does it pass?"*

Wait for the reviewer's response before moving to the next section. Options:
- **"Pass"** — move to next section
- **"Edit: [text]"** — note the edit, move on (reviewer will make the inline edit in GitHub)
- **"Redraft: [guidance]"** — draft a revised version of the section based on the guidance, present for approval

After all sections are reviewed, summarize:
> *"Review complete. [N] sections passed, [N] sections flagged for edits, [N] sections need redrafting.*
>
> *Next steps:*
> *1. Make any inline edits directly in the PR*
> *2. When satisfied, approve the PR (two approvals required before merge)*
> *3. After merge, the promote-template workflow will move the file to /approved/ and create the archive record"*

---

## Approval Checklist

Before recommending approval, confirm all items:

- [ ] All AMENDED sections accurately reflect documentation requirements for this pattern and algorithm
- [ ] All NET NEW sections are necessary, complete, and consistent with existing approved template standards
- [ ] Regulatory alignment confirmed — SR 26-2 (April 2026) reflected where applicable
- [ ] No INHERITED section has been unintentionally modified
- [ ] Document header updated to remove CANDIDATE status (or flagged for post-approval update)
- [ ] Template is ready to serve as a few-shot reference for future MDDs
- [ ] Two distinct MRM reviewers will approve — a single approval is never sufficient
